## SLIDE 3: Known Limitations

```
⚠️ KNOWN LIMITATIONS (MVP)

What We Have Now:
✅ Working conversational AI agent
✅ Real AWS Bedrock integration (Claude + Llama)
✅ Beautiful, responsive UI
✅ REST API backend
✅ Session management & conversation memory

What We're Using Temporarily:
⏳ Mock analytics data (420 simulated articles)
   → Phase 2: Connect real HubSpot, Google Analytics, WordPress
   
⏳ Single workspace only
   → Phase 2: Multi-user, multi-workspace support
   
⏳ No user authentication
   → Phase 2: SSO, role-based access control
   
⏳ In-memory session storage
   → Phase 2: Persistent database (PostgreSQL)

Performance Notes:
⏳ Response latency: 1-3 seconds (normal for first call)
   → Phase 2: <500ms with caching layer

⏳ No Slack integration yet
   → Phase 3: Slack bot for in-channel queries

⏳ No scheduled reports yet
   → Phase 3: Automated email/Slack delivery

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Why We Built It This Way:

"Show, don't tell"
├─ Built a working prototype to validate the core idea
├─ Proves conversational AI + analytics works
├─ Proves market demand exists
└─ De-risks the full product investment

MVP Focus:
├─ Minimal viable product validates the concept
├─ Mock data = no 3rd-party API dependencies
├─ Fast feedback loop with potential customers
├─ Build & learn before massive engineering investment

Real Data Comes Next:
├─ Phase 2 connects HubSpot, Google Analytics
├─ No architectural changes needed
├─ Data layer is modular & swappable
└─ Current code is production-ready

Security & Scalability Roadmap:
✅ Input validation & CORS config (security)
✅ Prepared for AWS Lambda deployment (scalability)
✅ Stateless design (horizontal scaling)
✅ Session management ready for database migration

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Technical Debt Acknowledged & Planned:

1. Session Persistence
   Current: In-memory (lost on server restart)
   Fix: PostgreSQL for durability (Phase 2)

2. Response Times
   Current: 1-3 seconds (model warm-up)
   Fix: Caching layer + Lambda optimization (Phase 2)

3. Conversation History Limits
   Current: Last 20 messages per session
   Fix: Unlimited history with database (Phase 2)

Bottom Line:
This MVP proves the concept works. Phase 2 turns it into
a production system ready for paying customers.
```

### **Presenter Notes:**

> "I want to be transparent about what's MVP and what's coming. Right now, we're using mock data. That was intentional—I wanted to prove the core idea works before spending weeks on data integrations. But the architecture is modular. Phase 2, we flip a switch and connect real HubSpot and Google Analytics data. No code refactoring needed.
>
> Same with user management. We don't have multi-workspace support yet, but the backend is ready for it. Same with persistence. We're storing sessions in memory right now—they disappear when the server restarts. Phase 2, we move to PostgreSQL for durability.
>
> Why did I build it this way? Because I wanted to validate the core assumption: does conversational AI actually help content teams? The answer is yes. Now that we know that works, Phase 2 is all about making it production-ready and connecting real data.
>
> I'm also aware of technical debt. Response times are 1-3 seconds. That's normal, but with caching and Lambda optimization, we can get to under 500ms. Conversation history currently caps at 20 messages. Phase 2, unlimited. These aren't breaking issues—they're just things we optimize as we scale.
>
> The bottom line: this MVP validates the product-market fit. Phase 2 turns it into a system ready for paying customers."
